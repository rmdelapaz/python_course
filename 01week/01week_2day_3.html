Introduction to Remote Repositories

So far, we've focused on local Git repositories. However, much of Git's power comes from its ability to synchronize with remote repositories, enabling collaboration and providing backup.

A remote repository is a version of your project hosted on the internet or network somewhere. It could be:

  • A repository hosting service like GitHub, GitLab, or Bitbucket
  • A server within your organization
  • Another computer on your local network

Working with remote repositories allows you to:

  • Collaborate with other developers on the same project
  • Backup your code to a secure location
  • Share your work with the community
  • Contribute to open-source projects
  • Deploy code to production environments

The Distributed Nature of Git

Unlike centralized version control systems where there's one canonical repository, Git is distributed—every developer has their own complete copy of the repository. This means:

  • You can work offline and commit changes locally
  • There's no single point of failure
  • Multiple backup copies of the repository exist naturally
  • You have the freedom to use various collaboration workflows

Remote repositories act as hubs for sharing and synchronizing work, but they're not fundamentally different from local repositories—they're just stored in a different location.

What is GitHub?

GitHub is the most popular platform for hosting Git repositories. It's a web-based service that provides:

  • A central location to store your code
  • Tools for code review and project management
  • Issue tracking and project planning
  • Documentation hosting
  • Social features to discover and contribute to other projects
  • CI/CD (Continuous Integration/Continuous Deployment) capabilities

Think of GitHub as a social network for code, where developers can share projects, collaborate with others, and build their professional portfolios.

GitHub Alternatives

While GitHub is the most widely used Git hosting platform, especially for open-source projects, alternatives include:

  • GitLab: Offers both cloud-hosted and self-hosted options with integrated CI/CD
  • Bitbucket: Popular for integration with other Atlassian products like Jira
  • Azure DevOps: Microsoft's solution integrating with their wider ecosystem
  • SourceForge: One of the oldest code hosting platforms
  • Self-hosted options: Like Gitea or GitLab Community Edition

In this course, we'll focus on GitHub as it's the most widely used platform, but most concepts apply to other Git hosting services as well.

GitHub vs. Git

It's important to understand the distinction between Git and GitHub:

  • Git is the distributed version control system—a tool installed on your computer that manages changes to your code
  • GitHub is a web-based hosting service for Git repositories with additional collaboration features

Think of it this way: Git is the tool, and GitHub is a service built around that tool. You can use Git without GitHub, but you can't use GitHub without Git.

Creating a GitHub Account

If you haven't already, let's set up a GitHub account. This will be your identity in the GitHub community and where you'll host your code projects.

Step-by-Step Account Creation

  1. Visit github.com
  2. Click "Sign up" in the top-right corner
  3. Enter your email address
  4. Create a password (use a strong, unique password or a password manager)
  5. Choose a username—this will be your GitHub identity and part of your repository URLs
    • Can contain alphanumeric characters and hyphens
    • Cannot start or end with a hyphen
    • Choose something professional, as it will be visible on your projects
  6. Answer the verification questions
  7. Click "Create account"
  8. Verify your email address when you receive the confirmation email

Setting Up Your GitHub Profile

Your GitHub profile is your developer identity. To make it professional and informative:

  1. Click your profile picture in the top-right corner and select "Your profile"
  2. Click "Edit profile" to add:
    • A professional profile picture
    • Your full name
    • A short bio describing your skills and interests
    • Your location (if you're comfortable sharing)
    • A website or portfolio link
    • Company/organization (if applicable)
  3. Consider creating a special repository named exactly the same as your username (e.g., username/username) to add a README to your profile page

Security Best Practices

Protect your GitHub account with these security measures:

  • Enable two-factor authentication (2FA): This adds a second layer of security beyond your password
  • Use SSH keys instead of passwords for repository access (we'll cover this later)
  • Regularly review authorized applications that have access to your account
  • Never commit sensitive information like passwords or API keys to your repositories

Creating a Remote Repository

Now that you have a GitHub account, let's create your first remote repository to host your code.

Creating a New Repository on GitHub

  1. Log in to GitHub
  2. Click the "+" icon in the top-right corner and select "New repository"
  3. Fill in the repository details:
    • Repository name: Choose a descriptive name (e.g., "python_calculator")
    • Description: Add a short description of your project
    • Visibility: Choose Public or Private
      • Public: Anyone can see the repository, but you control who can commit
      • Private: You choose who can see and commit to the repository
    • Initialize with README: Check this if starting from scratch
    • Add .gitignore: Select a template (e.g., Python) to automatically add common ignore patterns
    • Add a license: Choose an appropriate license for your project
  4. Click "Create repository"

Repository Visibility

When deciding between public and private repositories:

  • Public repositories are good for:
    • Open-source projects
    • Portfolio projects to showcase your skills
    • Learning projects where you might want community help
    • Code you want to share with the community
  • Private repositories are better for:
    • Commercial or proprietary code
    • Client work or company projects
    • Projects with sensitive information
    • Works in progress you're not ready to share

Remember: Even with private repositories, GitHub staff technically have access to your code, so never store sensitive credentials in any repository.

Choosing a License

Licenses define how others can use, modify, and distribute your code. Common choices include:

  • MIT License: Very permissive, allows commercial use, modification, distribution, and private use
  • GNU General Public License (GPL): Requires derivative works to also be open-source
  • Apache License 2.0: Similar to MIT but with patent protections
  • BSD Licenses: Family of permissive licenses with various requirements

For this course, MIT or Apache 2.0 are good choices for most projects. If you're unsure, choosealicense.com can help you decide.

Connecting Local and Remote Repositories

Now that we have both a local Git repository and a remote GitHub repository, let's connect them.

Scenario 1: Existing Local Repository, New Remote Repository

If you already have a local repository that you want to push to a new GitHub repository:

# First, create a new repository on GitHub without initializing it
# Then, connect your local repository to the remote
git remote add origin https://github.com/yourusername/repository-name.git

# Verify the remote was added
git remote -v

# Ensure your main branch is named correctly
git branch -M main

# Push your local repository to GitHub
git push -u origin main

The -u (or --set-upstream) flag sets the upstream, linking your local branch to the remote branch. This means in the future, you can simply use git push and git pull without specifying the remote and branch.

Scenario 2: New Local Repository from Existing Remote Repository

If you want to start with an existing GitHub repository:

# Clone the repository to your local machine
git clone https://github.com/yourusername/repository-name.git

# This creates a directory named "repository-name" with the repository content
cd repository-name

# Now you can work with the repository normally

Cloning automatically sets up the remote connection, so you don't need to add it manually.

Scenario 3: Starting from Scratch with GitHub

If you're starting a new project from scratch:

  1. Create a new repository on GitHub with initialization options (README, .gitignore, license)
  2. Clone the repository to your local machine: 
    git clone https://github.com/yourusername/repository-name.git
  3. Add your project files to the local repository
  4. Commit and push your changes

Understanding Remote References

When you connect to a remote repository, Git creates references to track the state of branches on that remote:

  • origin/main refers to the state of the main branch on the remote named origin
  • These references are updated when you fetch or pull from the remote
  • You can view remote branches with git branch -r

By convention, the primary remote repository is usually named "origin", but you can have multiple remotes with different names.

Working with Remote Repositories

Once you've connected local and remote repositories, you'll need to know how to synchronize changes between them.

Pushing Changes

To send your local commits to the remote repository:

# Push the current branch to its upstream branch
git push

# If upstream isn't set, or to push to a different branch
git push origin branch-name

# Push all branches
git push --all origin

Common scenarios:

  • First push of a new branch: git push -u origin branch-name
  • Regular push of existing branch: git push
  • Force push (use with caution): git push --force

Fetching and Pulling Changes

To get changes from the remote repository:

# Fetch updates from remote without merging
git fetch origin

# Pull (fetch + merge) changes from remote
git pull origin branch-name

# Pull with rebase instead of merge
git pull --rebase origin branch-name

The difference between fetch and pull:

  • git fetch: Downloads changes from the remote but doesn't apply them to your working directory—it just updates the remote tracking branches
  • git pull: Combines git fetch and git merge (or git rebase with the --rebase flag)—it downloads changes and integrates them into your working branch

Tracking Remote Branches

To set up a local branch to track a remote branch:

# When creating a new branch
git checkout -b feature-branch origin/feature-branch

# For an existing branch
git branch -u origin/feature-branch

# Another way to set upstream
git branch --set-upstream-to=origin/feature-branch feature-branch

Viewing Remote Information

To get information about remote repositories:

# List remote repositories
git remote -v

# Show information about a remote
git remote show origin

# List remote branches
git branch -r

# List both local and remote branches
git branch -a

GitHub Workflow

Let's explore a typical workflow for using GitHub in your development process.

Complete GitHub Workflow Example

Here's a step-by-step workflow for a typical feature development cycle:

1. Start with a Fresh Repository

# Clone the repository
git clone https://github.com/yourusername/project.git
cd project

2. Create a Feature Branch

# Create and switch to a new branch
git checkout -b feature/user-authentication

3. Make Changes and Commit

# Edit files...
# Stage changes
git add user_auth.py templates/login.html
# Commit changes
git commit -m "Implement basic user login functionality"

# Make more changes...
git add user_auth.py models.py
git commit -m "Add user model and authentication backend"

4. Push Your Branch to GitHub

# First push, set upstream
git push -u origin feature/user-authentication

5. Keep Your Branch Updated (Optional)

# Switch to main
git checkout main
# Get latest changes
git pull
# Switch back to your feature branch
git checkout feature/user-authentication
# Merge main into your branch
git merge main

6. Finish Your Feature and Push Final Changes

# Make final changes...
git add templates/register.html
git commit -m "Add user registration template"
# Push changes
git push

7. Create a Pull Request on GitHub

  1. Go to the repository page on GitHub
  2. Click "Pull requests" and then "New pull request"
  3. Select your feature branch to compare with main
  4. Click "Create pull request"
  5. Add a title and description
  6. Assign reviewers if applicable
  7. Submit the pull request

8. Address Review Comments

# Make changes based on review feedback
git add fixed_file.py
git commit -m "Address review feedback: fix validation bug"
git push

9. Merge the Pull Request

Once approved, merge the pull request on GitHub.

10. Clean Up

# Switch to main
git checkout main
# Pull the merged changes
git pull
# Delete the local feature branch
git branch -d feature/user-authentication
# Delete the remote feature branch (optional)
git push origin --delete feature/user-authentication

Key GitHub Concepts

Pull Requests

Pull requests (PRs) are GitHub's way of proposing changes and requesting review before merging branches. They provide:

  • A place to discuss code changes
  • Automated checks (if configured)
  • Line-by-line code review capabilities
  • Protected branches and required reviews

Issues

GitHub Issues are a way to track bugs, features, and other tasks:

  • Create issues to describe needed work
  • Assign issues to team members
  • Reference issues in commits and pull requests
  • Close issues automatically with keywords in commit messages

GitHub Actions

GitHub's built-in CI/CD platform allows you to automate workflows like:

  • Running tests when code is pushed
  • Building and deploying applications
  • Checking code style and quality
  • Publishing packages

Authentication with GitHub

When working with GitHub, you'll need to authenticate to push changes. GitHub offers two main authentication methods:

HTTPS Authentication

Using HTTPS is simple but requires entering your credentials:

  • Repository URLs look like: https://github.com/username/repository.git
  • You'll need to enter your GitHub username and password/token for operations like push
  • Since August 2021, GitHub no longer accepts passwords for Git operations
  • You must use a personal access token instead

Setting Up a Personal Access Token

  1. Go to GitHub Settings → Developer settings → Personal access tokens → Generate new token
  2. Select appropriate scopes (usually "repo" for full repository access)
  3. Generate the token and copy it immediately (it won't be shown again)
  4. Use this token in place of a password when pushing
  5. To avoid entering it repeatedly, you can configure credential caching: 
    # Cache credentials for 15 minutes
git config --global credential.helper cache

# Cache credentials for longer (e.g., 1 hour)
git config --global credential.helper "cache --timeout=3600"

SSH Authentication

SSH authentication is more secure and convenient for regular use:

  • Repository URLs look like: git@github.com:username/repository.git
  • Uses SSH key pairs for authentication
  • No need to enter a password for each operation
  • More secure than password authentication

Setting Up SSH Authentication

  1. Check for existing SSH keys: 
    ls -la ~/.ssh
  2. Generate a new SSH key if needed: 
    ssh-keygen -t ed25519 -C "your_email@example.com"
  3. Start the SSH agent: 
    eval "$(ssh-agent -s)"
  4. Add your private key to the SSH agent: 
    ssh-add ~/.ssh/id_ed25519
  5. Copy your public key to the clipboard: 
    # On macOS
pbcopy < ~/.ssh/id_ed25519.pub

# On Windows (PowerShell)
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard

# On Linux
xclip -selection clipboard < ~/.ssh/id_ed25519.pub
# or
cat ~/.ssh/id_ed25519.pub
  6. Add the SSH key to your GitHub account:
    1. Go to GitHub Settings → SSH and GPG keys → New SSH key
    2. Add a title (e.g., "Work Laptop")
    3. Paste your public key
    4. Click "Add SSH key"
  7. Test your SSH connection: 
    ssh -T git@github.com

Switching Between HTTPS and SSH

To change the remote URL for an existing repository:

# From HTTPS to SSH
git remote set-url origin git@github.com:username/repository.git

# From SSH to HTTPS
git remote set-url origin https://github.com/username/repository.git

Verify the change with:

git remote -v

Practical Exercise: GitHub Workflow

Let's put what we've learned into practice with a hands-on exercise:

Exercise: Set Up and Use a GitHub Repository

Part 1: Create a GitHub Repository

  1. Go to GitHub and log in to your account
  2. Click the "+" icon in the top-right corner and select "New repository"
  3. Name the repository "python_calculator" (or use the name from our previous exercise)
  4. Add a description: "A simple calculator implemented in Python"
  5. Choose "Public" visibility
  6. Check "Add a README file"
  7. Choose "Python" from the "Add .gitignore" dropdown
  8. Choose "MIT License" from the "Add a license" dropdown
  9. Click "Create repository"

Part 2: Clone the Repository

# Navigate to your desired parent directory
cd ~/projects

# Clone the repository
git clone https://github.com/yourusername/python_calculator.git

# Navigate into the repository
cd python_calculator

# Verify repository contents
ls -la

Part 3: Add Calculator Code

# Create the calculator file
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b

def multiply(a, b):
    return a * b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

def power(a, b):
    return a ** b

if __name__ == "__main__":
    print("Simple Calculator")
    print("10 + 5 =", add(10, 5))
    print("10 - 5 =", subtract(10, 5))
    print("10 * 5 =", multiply(10, 5))
    print("10 / 5 =", divide(10, 5))
    print("10 ^ 2 =", power(10, 2))
EOF

Part 4: Commit and Push Changes

# Check status
git status

# Stage the new file
git add calculator.py

# Commit changes
git commit -m "Add basic calculator functions"

# Push to GitHub
git push origin main

Part 5: Create a Feature Branch

# Create and checkout a new branch
git checkout -b feature/modulo-function

# Add a new function to calculator.py
cat >> calculator.py << 'EOF'

def modulo(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a % b

if __name__ == "__main__":
    # Add this line to the existing main block
    print("10 % 3 =", modulo(10, 3))
EOF

# Stage and commit the changes
git add calculator.py
git commit -m "Add modulo function"

# Push the feature branch to GitHub
git push -u origin feature/modulo-function

Part 6: Create a Pull Request

  1. Go to your repository on GitHub
  2. You should see a notification about your recently pushed branch
  3. Click "Compare & pull request"
  4. Set the title to "Add modulo function"
  5. Add a description: "This PR adds a modulo function to calculate remainders"
  6. Click "Create pull request"

Part 7: Review and Merge the Pull Request

  1. On the pull request page, review the changes
  2. Click on the "Files changed" tab to see the code diff
  3. Add a comment on a specific line of code (optional)
  4. Go back to the "Conversation" tab
  5. Click "Merge pull request" and confirm

Part 8: Update Your Local Repository

# Switch back to the main branch
git checkout main

# Pull the merged changes
git pull origin main

# Verify the changes are in your local main branch
cat calculator.py

# Delete the feature branch (optional)
git branch -d feature/modulo-function

Bonus: Add GitHub Actions for Testing

  1. On GitHub, navigate to the "Actions" tab of your repository
  2. Select the "Python Application" workflow template
  3. Commit the workflow file to your repository
  4. Create a test file locally: 
    mkdir tests
cat > tests/test_calculator.py << 'EOF'
import unittest
from calculator import add, subtract, multiply, divide, power, modulo

class TestCalculator(unittest.TestCase):
    def test_add(self):
        self.assertEqual(add(10, 5), 15)
        self.assertEqual(add(-1, 1), 0)
        
    def test_subtract(self):
        self.assertEqual(subtract(10, 5), 5)
        self.assertEqual(subtract(5, 10), -5)
        
    def test_multiply(self):
        self.assertEqual(multiply(10, 5), 50)
        self.assertEqual(multiply(0, 5), 0)
        
    def test_divide(self):
        self.assertEqual(divide(10, 5), 2)
        self.assertEqual(divide(10, 2), 5)
        with self.assertRaises(ValueError):
            divide(10, 0)
            
    def test_power(self):
        self.assertEqual(power(10, 2), 100)
        self.assertEqual(power(2, 3), 8)
        
    def test_modulo(self):
        self.assertEqual(modulo(10, 3), 1)
        self.assertEqual(modulo(10, 5), 0)
        with self.assertRaises(ValueError):
            modulo(10, 0)

if __name__ == '__main__':
    unittest.main()
EOF
  5. Push the test file to GitHub: 
    git add tests/test_calculator.py
git commit -m "Add calculator unit tests"
git push origin main

    6. Then enable GitHub Pages in your repository settings, selecting the "docs" folder as the source.

Summary and Additional Resources

Key Concepts Recap

  • Remote repositories allow you to backup and share your code
  • GitHub provides hosting for Git repositories with additional features
  • Authentication can be done via HTTPS with tokens or SSH keys
  • Collaboration workflows include direct collaboration and fork-and-pull models
  • Pull requests facilitate code review and discussion
  • Merge conflicts are a normal part of collaboration and can be resolved
  • Tags and releases mark important points in your project's history
  • GitHub Pages provides free hosting for static websites

Additional GitHub Features

  • GitHub Projects: Project management boards integrated with your repository
  • GitHub Codespaces: Cloud development environments
  • GitHub Copilot: AI-powered code completion

Best Practices to Remember

  • Document your projects with clear READMEs and appropriate documentation
